## 2. Layout Architecture

### Root Layout (`app/layout.tsx`)

The root layout is a Server Component (async). It provides:

- **HTML wrapper** with `lang="en"`, `dir="ltr"`, font CSS variable class (`fontsVariable`)
- **Skip-to-content link** for keyboard accessibility
- **RealViewport** -- sets CSS custom properties for viewport dimensions (`--vw`, `--dvh`)
- **TransformProvider** -- hierarchical transform context for WebGL coordinate mapping
- **OptionalFeatures** -- conditionally loads WebGL canvas, dev tools based on config
- **Sanity Visual Editing** -- only when draft mode is enabled
- **ReactTempus** -- RAF management, patching disabled in draft mode

Critical rule: **Header and Footer are NOT in the root layout.** They live inside the `<Wrapper>` component. The layout comment explicitly states this.

### Page Pattern (`app/page.tsx`)

Every page wraps its content in `<Wrapper>`:

```tsx
export default function Page() {
  return (
    <Wrapper theme="dark" lenis={{}}>
      {/* sections here */}
    </Wrapper>
  )
}
```

The current homepage imports sections from `app/(marketing)/_sections/`. These are demo content meant to be replaced.

### Wrapper Component (`components/layout/wrapper/index.tsx`)

The Wrapper is the central layout orchestrator. It renders in this order:

1. `<Theme>` -- applies `data-theme` attribute to `<html>`, provides theme context
2. `<Header />` -- fixed navigation (rendered BY Wrapper, not layout)
3. `<Canvas>` -- WebGL context wrapper (activates GlobalCanvas when `webgl={true}`)
4. `<main id="main-content">` -- page content slot with `className="relative flex grow flex-col"`
5. `<Footer />` -- site footer (rendered BY Wrapper, not layout)
6. `<Lenis>` -- smooth scroll provider (when `lenis` prop is truthy)

Props:
- `theme` -- `'dark' | 'light' | 'red' | 'evil'` (default: `'dark'`)
- `lenis` -- `boolean | LenisOptions` (default: `true`)
- `webgl` -- `boolean` (default: `false`)
- `className` -- passed to `<main>`

## 3. Theme System

### Colors (`lib/styles/colors.ts`)

Base palette:
```
black:  #000000
white:  #ffffff
red:    #e30613
blue:   #0070f3
green:  #00ff88
purple: #7928ca
pink:   #ff0080
```

### Theme Definitions

Each theme maps three semantic tokens: `primary`, `secondary`, `contrast`.

| Theme | Primary (bg) | Secondary (text) | Contrast (accent) |
|-------|-------------|-----------------|-------------------|
| `light` | white | black | red |
| `dark` | black | white | red |
| `evil` | black | red | white |
| `red` | red | black | white |

Themes are applied via `data-theme` attribute on `<html>`. The `<Theme>` component sets this attribute and provides a React context.

### CSS Variables

Theme colors are available as Tailwind theme colors AND CSS custom properties:
- `--color-primary`, `--color-secondary`, `--color-contrast` -- semantic, theme-aware
- `--color-black`, `--color-white`, `--color-red`, etc. -- static palette

Usage: `bg-primary`, `text-secondary`, `border-contrast` in Tailwind, or `var(--color-primary)` in CSS.

### Theme Variants in Tailwind

Custom variants for theme-conditional styling:
```css
@custom-variant light (&:where([data-theme=light], [data-theme=light] *));
@custom-variant dark  (&:where([data-theme=dark], [data-theme=dark] *));
@custom-variant evil  (&:where([data-theme=evil], [data-theme=evil] *));
@custom-variant red   (&:where([data-theme=red], [data-theme=red] *));
```

Usage: `light:text-black dark:text-white`

### useTheme Hook

```tsx
const { state, actions } = useTheme()
// state.name: ThemeName
// state.theme: { primary, secondary, contrast }
// actions.setTheme: (theme: ThemeName) => void
```

## 4. Design Tokens & CSS Variables

### Layout Variables (`lib/styles/css/root.css`)

Generated by `setup-styles.ts` -- do NOT edit directly.

| Variable | Mobile | Desktop |
|----------|--------|---------|
| `--device-width` | 375 | 1440 |
| `--device-height` | 650 | 816 |
| `--columns` | 4 | 12 |
| `--gap` | ~4.27vw (16px/375) | ~1.11vw (16px/1440) |
| `--safe` | ~4.27vw (16px/375) | ~1.11vw (16px/1440) |
| `--header-height` | ~15.47vw (58px/375) | ~6.81vw (98px/1440) |

Derived variables:
- `--layout-width`: `calc(100vw - (2 * var(--safe)))`
- `--column-width`: `calc((var(--layout-width) - (var(--columns) - 1) * var(--gap)) / var(--columns))`

### Breakpoints

Single breakpoint: **800px** (desktop threshold).

```css
@custom-media --mobile (width <= 799.98px);
@custom-media --desktop (width >= 800px);
```

Tailwind breakpoint: `dt:` prefix for desktop.

### Easing Functions

18 easing CSS custom properties available: `--ease-in-quad` through `--ease-in-out-circ`, plus `--ease-gleasing`.

### Custom Spacing

`--spacing-safe`, `--spacing-gap`, `--spacing-header-height` map to the layout variables. Usable as Tailwind spacing: `p-safe`, `gap-gap`, `pt-header-height`.

## 5. Responsive Scaling System (`dr-*` Utilities)

The template uses a design-pixel-to-viewport scaling system. All `dr-*` utilities convert a pixel value from the design comp into a responsive `vw` unit.

Formula: `calc((value * 100) / var(--device-width) * 1vw)`

On mobile (`--device-width: 375`), `dr-w-100` = `calc(100 * 100 / 375 * 1vw)` = ~26.67vw.
On desktop (`--device-width: 1440`), `dr-w-100` = `calc(100 * 100 / 1440 * 1vw)` = ~6.94vw.

Available `dr-*` utilities cover every CSS property:
- **Size**: `dr-w-*`, `dr-h-*`, `dr-min-w-*`, `dr-max-w-*`, `dr-min-h-*`, `dr-max-h-*`
- **Spacing**: `dr-p-*`, `dr-px-*`, `dr-py-*`, `dr-pt/pr/pb/pl-*`, `dr-m-*`, `dr-mx/my/mt/mr/mb/ml-*`
- **Position**: `dr-top-*`, `dr-right-*`, `dr-bottom-*`, `dr-left-*`, `dr-inset-*`
- **Gap**: `dr-gap-*`, `dr-gap-x-*`, `dr-gap-y-*`
- **Typography**: `dr-text-*`, `dr-tracking-*`, `dr-leading-*`
- **Border**: `dr-border-*`, `dr-rounded-*` (all sides/corners)

### Column-Based Sizing (`dr-*-col-*`)

Size elements by grid columns: `dr-w-col-4` = width of 4 columns + 3 gaps.

### Layout Utilities

- `dr-grid` -- grid with `--columns` columns and `--gap`
- `dr-layout-block` -- centered block with `--safe` margin
- `dr-layout-block-inner` -- padded block with `--safe` padding
- `dr-layout-grid` -- `dr-layout-block` + `dr-grid`
- `dr-layout-grid-inner` -- `dr-layout-block-inner` + `dr-grid`
- `desktop-only` / `mobile-only` -- responsive visibility

## 12. Enforced Rules (Build-Breaking)

1. **Use `Image` from `@/components/ui/image`** -- never `next/image` or `<img>`. Biome rule `noImgElement: error`.
2. **Use `Link` from `@/components/ui/link`** -- never `next/link` or `<a>`. Biome plugin `no-anchor-element`.
3. **CSS Modules imported as `s`** -- `import s from './component.module.css'`
4. **Path aliases required** -- no `../` imports. Biome plugin `no-relative-parent-imports`.
5. **No manual memoization** -- React Compiler handles it. No `useMemo`, `useCallback`, `React.memo`.
6. **No `any`** -- `noExplicitAny: error`. Use `unknown` + type narrowing.
7. **Sorted Tailwind classes** -- `useSortedClasses: error` in `className`, `cn()`, `clsx()`.
8. **`import type` for type-only imports** -- `verbatimModuleSyntax: true`, `useImportType: error`.
9. **Server Components by default** -- only add `'use client'` when needed.
10. **No `forwardRef`** -- React 19, `ref` is a regular prop.

## 15. Component Conventions

### Standard Pattern

```tsx
'use client' // only if needed

import cn from 'clsx'
import type { ComponentProps } from 'react'
import s from './my-component.module.css'

interface MyComponentProps extends ComponentProps<'div'> {
  variant?: 'primary' | 'secondary'
}

export function MyComponent({ variant = 'primary', className, ...props }: MyComponentProps) {
  return <div className={cn(s.root, className)} {...props} />
}
```

### Naming
- **Files**: kebab-case (`my-component.tsx`, `my-component.module.css`)
- **CSS classes**: camelCase (`.isPrimary`, `.isDisabled`)
- **Components**: PascalCase function declarations (not arrow functions)
- **Exports**: named exports, not default (except in `app/` directory)

### Compound Components

Wrap @base-ui/react primitives. Provide both simple API and compound API. CSS via modules as `s`, merged with `cn()`. Always pass `className` through.

### Context Pattern

Use `createStandardContext` for shared state. Shape: `{ state, actions, meta? }`.

## 16. What Must Be Customized for StarkZap

### Must Replace
- `app/page.tsx` -- replace demo homepage content
- `app/(marketing)/_sections/` -- delete or replace entirely
- `components/layout/header/index.tsx` -- replace nav links, branding, layout
- `components/layout/footer/index.tsx` -- replace branding and links
- `lib/styles/fonts.ts` -- add project fonts
- `lib/styles/typography.ts` -- add typography presets
- `lib/styles/colors.ts` -- update color palette and themes
- `package.json` -- update name, description

### Must Preserve (Framework Code)
- `components/layout/wrapper/index.tsx` -- use as-is, it orchestrates everything
- `components/ui/*` -- all UI primitives, use without modification
- `lib/hooks/*` -- all hooks
- `lib/utils/*` -- all utilities
- `lib/styles/css/root.css` -- generated file, do not edit (run `bun setup:styles`)
- `lib/styles/css/tailwind.css` -- generated file, do not edit
- `lib/styles/css/global.css` -- can extend but avoid modifying existing rules
- Root layout structure (`app/layout.tsx`) -- preserve provider hierarchy

### Can Extend
- `lib/styles/colors.ts` -- add new colors, modify themes
- `lib/styles/layout.mjs` -- adjust breakpoints, grid, safe areas
- `lib/styles/typography.ts` -- add typography presets
- `biome.json` -- add rules, keep existing ones
- `next.config.ts` -- add remote image patterns, redirects, rewrites
